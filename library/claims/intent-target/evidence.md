# Evidence Capture

Capture evidence for this claim type in the shared evidence shape: claim,
context or method, observations and raw artifacts, verdict.

## Claim

- State the concrete claim using the rules in `claim.md`.
- Name the intended confusion reduction, audience, task, and success target.
- Include any non-goal that limits how success should be interpreted.

## Methodology References

- Record the claim type folder path.
- Link the relevant `plan.md` intent and intake sections.
- Link any receipt, decision-ledger entry, or owner instruction that sharpened
  the target.

## Method Or Context

- Explain how the intended target was established.
- Record whether the target came from explicit intake, clarification during the
  run, or a previously fixed decision.
- Note any change to the target and why it occurred.

## Observations

- Record the concrete statements that define the intended outcome.
- Capture evidence that the pass work aligned with that target.
- Record mismatches between the target and what the pass actually touched.
- Include any non-goals or exclusions that affect the meaning of success.

## Raw Artifacts

- Reference `plan.md` sections for intended outcome, audience, task, goals,
  non-goals, and done definition.
- Reference receipts that show actions taken in service of the target.
- Reference report excerpts if the pass later assesses whether the target was
  met.

## Verdict

- Use the verdict meanings defined below.
- Base the verdict on whether the pass target is explicit, stable enough, and
  reflected in the actual work.
- Note the limiting factor when the target remained ambiguous or shifted.

## Open Questions Or Follow-Ups

- Record unresolved ambiguity about audience, task, or success target.
- Record revisit triggers when the intended target changed mid-run.
- Include an owner when the pass cannot set the target unilaterally.

## Verdict Guidance

Define what each allowed verdict means for this claim type.

### supported

- Use when: the intended outcome, audience, task, and success target are
  explicit and the work performed aligns with them.
- Not enough when: the pass names an outcome but leaves success or audience
  implicit.

### mixed

- Use when: the target is mostly clear but one important element is weak, such
  as a vague success target or a late reframe.
- Not enough when: the core problem being reduced is still ambiguous.

### incomplete

- Use when: the run is clearly about something, but the audience, task, or
  success target is not yet explicit enough to support later trust.
- Not enough when: the target was contradicted by the work done.

### open

- Use when: the target claim is active but still awaiting clarification or
  approval.
- Not enough when: the run has already finalized work against an undefined
  target.

### failed

- Use when: the work performed materially diverged from the stated target or the
  target claim was shown to be incorrect.
- Not enough when: only minor wording drift exists with no impact on the pass
  direction.

## Evidence Thread Rules

Create a new evidence item when:

- the pass objective changes materially,
- the primary audience or task changes materially,
- one thread would otherwise mix distinct outcomes.

### Enrich An Existing Evidence Item When

- clarifications sharpen the same target,
- more observations show whether execution stayed aligned with the same target,
- the verdict changes as the pass progresses against the same target.

### Do Not Mix

- multiple independent pass goals,
- target definition with diagnosis of concrete wiki problems,
- target definition with approval or ownership evidence that belongs in
  `governance-ownership`.

### Close Out When

- the pass target is stable for the current run,
- the verdict captures whether the run aligned with it,
- remaining ambiguity is recorded explicitly.
